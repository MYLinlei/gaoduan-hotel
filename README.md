# 高端酒店品牌后端改造说明书

## 1. 项目定位

本项目最终目标不是同时维护 `sky-take-out` 和 `hm-dianping` 两个独立项目，而是：

- 以 `sky-take-out` 作为唯一主工程
- 以 `hm-dianping` 作为参考代码仓库
- 在 `sky-take-out` 上完成二次开发
- 落地为一个单酒店、纯私域、高端餐饮品牌后端系统

最终业务形态：

- 单一高端酒店品牌
- 自有官网用户端
- 堂食 / 外卖双模式
- 高端菜品 + 名酒
- 大额优惠券抢购 / 秒杀
- 自有骑手配送
- 菜品种草、评价、点赞、收藏

一句话概括：

`苍穹外卖做交易主干，黑马点评做能力参考，最终沉淀为高端酒店私域后端系统。`

## 4. 总体改造原则

### 4.1 必须坚持的原则

- 所有正式开发都在 `sky-take-out` 中完成
- `hm-dianping` 仅作参考，不作主项目
- 不推翻现有 `sky-take-out` 架构
- 优先补齐链路，再叠加高级功能
- 先改后端模型，再改前端页面
- 所有新功能围绕“单酒店”建模，不做多商户扩展

### 4.2 暂时不要做的事

- 不要先做官网页面
- 不要先做 GEO 删除
- 不要先做大规模重命名
- 不要先把黑马点评整包复制
- 不要一开始就拆微服务

## 5. 标准开发顺序

后续开发必须按下面顺序推进。

### 第 1 阶段：补齐 `sky-take-out` 用户交易链路基线

目标：

- 让主工程重新具备完整的用户下单闭环

需要补的模块：

- 购物车
- 地址簿
- 用户订单提交
- 用户订单查询
- 用户取消订单
- 支付流程占位

这一阶段完成标准：

- 用户可以完成从浏览菜品到提交订单的最小闭环
- 后台可以看到新订单
- 后台订单状态可以正常流转

### 第 2 阶段：输出酒店定制数据库 DDL

目标：

- 把高端酒店业务模型一次性设计清楚

重点改造：

- `dish` 增加高端属性字段
- `orders` 增加堂食 / 外卖区分字段
- 新增高端优惠券相关表
- 新增菜品种草 / 评价 / 点赞 / 收藏相关表
- 新增骑手表与配送表

这一阶段完成标准：

- 所有新增表和新增字段落库
- 实体类、DTO、Mapper 同步完成第一版定义

### 第 3 阶段：改订单状态机，拆堂食 / 外卖

目标：

- 从单一外卖订单流，升级为双订单模式

建议状态设计：

- 堂食：待支付 -> 已支付 -> 待制作 -> 制作中 -> 已上桌 -> 已完成 -> 已取消
- 外卖：待支付 -> 已支付 -> 待接单 -> 已接单 -> 待取餐 -> 配送中 -> 已送达 -> 已完成 -> 已取消

关键要求：

- 堂食订单不进入骑手派单链路
- 外卖订单才进入骑手派单链路
- 后台和后厨页面都要按订单类型区分展示

### 第 4 阶段：接入高端优惠券与秒杀

目标：

- 把黑马点评的高并发能力迁移到主项目

只迁这些能力：

- Redis 预扣库存
- Lua 原子校验
- RedisIdWorker
- Redisson 分布式锁
- 超卖防护

不直接复制的内容：

- 原始商铺业务代码
- 原始 ShopController / ShopService 全量逻辑

业务落地建议：

- 全店通用高端券
- 酒水专用券
- 指定菜品券
- 宴会专属券
- 每人每日限购

### 第 5 阶段：接入菜品种草 / 评价 / 点赞 / 收藏

目标：

- 把黑马点评的“商家维度互动”重建为“菜品维度互动”

建议拆分：

- `dish_note`：菜品种草 / 名酒品鉴笔记
- `dish_comment`：菜品评价
- `dish_like`：菜品点赞
- `dish_favorite`：菜品收藏

注意：

- 这里是业务重建，不是机械地把 `shop_id` 替换成 `dish_id`
- 推荐排序应基于销量、评分、点赞、收藏、评价权重综合计算

### 第 6 阶段：接入自有骑手与后台管理

目标：

- 让外卖订单只流向酒店内部骑手

核心要求：

- 骑手只能后台录入
- 不做骑手自主注册
- 不接第三方配送平台
- 配送区域固定，不做 GEO 定位计算

后台新增页面建议：

- 高端优惠券管理
- 堂食桌号管理
- 自有骑手管理
- 菜品种草审核
- 双订单模式管理页

### 第 7 阶段：官网前端接口与页面联调

目标：

- 上线独立用户端官网

官网页面建议：

- 品牌首页
- 菜品分类页
- 菜品详情页
- 名酒专区
- 高端优惠券页面
- 堂食 / 外卖下单页
- 我的订单页
- 我的收藏 / 优惠券页

官网详情页建议聚合：

- 菜品介绍
- 名酒品鉴笔记
- 用户评价
- 点赞 / 收藏
- 抢券入口
- 下单入口

## 6. 推荐数据库设计方向

### 6.1 需要扩展的核心表

#### `dish`

建议新增字段：

- `tag_type`：菜品标签类型，名酒 / 奢华餐食 / 宴会推荐
- `luxury_level`：轻奢等级
- `recommend_weight`：推荐权重
- `like_count`
- `favorite_count`
- `note_count`
- `comment_count`

#### `orders`

建议新增字段：

- `order_type`：1 外卖，2 堂食
- `table_no`
- `rider_id`
- `delivery_zone_code`
- `coupon_id`
- `coupon_amount`
- `actual_pay_amount`

### 6.2 建议新增的业务表

- `hotel_table`
- `hotel_rider`
- `hotel_high_voucher`
- `hotel_high_voucher_order`
- `dish_note`
- `dish_comment`
- `dish_like`
- `dish_favorite`

### 6.3 优惠券作用域设计建议

不要只绑定 `dish_id`，建议做成：

- `scope_type`：`ALL_STORE` / `CATEGORY` / `DISH`
- `scope_id`

这样后续才能支持：

- 全店券
- 酒水券
- 指定菜品券

## 7. 模块迁移映射关系

### 7.1 从 `hm-dianping` 借鉴到 `sky-take-out` 的内容

| 来源 | 去向 | 用法 |
|---|---|---|
| `CacheClient` | 菜品缓存模块 | 改造成 `dish` 维度缓存 |
| `RedisIdWorker` | 优惠券订单 / 活动单号 | 直接参考实现 |
| `seckill.lua` | 高端优惠券秒杀 | 改参数和 key 前缀 |
| `RedissonConfig` | 主工程配置 | 按主项目版本重新接入 |
| `VoucherOrderServiceImpl` 思路 | 酒店高端券秒杀服务 | 只迁架构，不原样照搬 |
| `Blog` 表意 | `dish_note` | 重新定义为菜品种草 |
| `BlogComments` 表意 | `dish_comment` | 重新定义为菜品评价 |

### 7.2 不要直接迁移的内容

- `ShopController`
- `ShopServiceImpl` 原始商铺查询逻辑
- `ShopType`
- GEO 附近门店能力
- 多商户相关语义

## 10. 不推荐的 AI 指令写法

以下写法容易让 AI 跑偏：

- “帮我把这两个项目融合一下”
- “你直接全做完”
- “把黑马点评功能全部迁过来”
- “先随便改改看”
- “把官网和后端一起做了”

原因：

- 任务过大
- 范围不清
- 主次不分
- 很容易让 AI 去做错误的架构决策
